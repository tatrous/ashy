## КАК ПОЛЬЗОВАТЬСЯ ШАБЛОНОМ

Это эталонная книга продукта Vibecoding Workshops. Она устроена так, чтобы продукт описывался **и как продукт, и как бизнес**.

**Три уровня глубины:**
- **[Core]** — обязательно для любого проекта. Без этого MVP не собрать.
- **[Business]** — обязательно, если хочешь смотреть на проект как на бизнес, а не учебную игрушку.
- **[Advanced]** — можно отложить до второй итерации. Помечено отдельно — пропуск не сломает основную логику.

**Заполняй сверху вниз.** Каждый следующий раздел опирается на предыдущий: рынок → аудитория → конкуренты → CJM → бизнес-модель → платформы → фичи → экраны → данные → стек → MVP → GTM → рост → метрики → риски.

**Если продукт мульти-канальный** (web + бот, mobile + расширение, и т.д.) — раздел 7 «Платформы продукта» становится центральным. Оттуда тянутся user stories, экраны и MVP-скоуп.

> **Базовое требование курса:** в финальном продукте курса обязательно должны быть **и web, и мобильное приложение** — это база, без которой курс не считается пройденным. Дополнительные платформы (Telegram-бот, расширения, desktop, voice) — опциональны и добавляются поверх базы как companion/acquisition.

## 3. РЫНОК И ЮНИТ-ЭКОНОМИКА [Business]

### 3.1 Размер рынка (TAM / SAM / SOM)

> Это не «попадание в Forbes», а проверка — есть ли вообще на чём зарабатывать. Считай грубо, указывай источники.

| Уровень | Что это | Расчёт | Цифра | Источник |
|---------|---------|--------|-------|----------|
| **TAM** (Total Addressable Market) | Весь мировой/региональный рынок этой проблемы | напр. «N млн человек × средний чек × частота» | | |
| **SAM** (Serviceable Available Market) | Та часть TAM, до которой реально дотянуться (язык, платформа, регион) | | | |
| **SOM** (Serviceable Obtainable Market) | Реалистичная доля SAM на горизонте 1–3 года | | | |

```

### 3.2 Юнит-экономика на одного пользователя

| Метрика | Значение | Как считал |
|---------|----------|------------|
| **ARPU** (средний доход на пользователя/мес) | | цена × конверсия в платящего |
| **Gross margin** (маржа после прямых расходов) | | (доход − стоимость инфры/AI/комиссий) ÷ доход |
| **CAC** (стоимость привлечения) | | бюджет канала ÷ кол-во привлечённых |
| **LTV** (life-time value) | | ARPU × gross margin × средний срок жизни (мес) |
| **LTV / CAC** | | здоровое — от 3:1 |
| **Payback period** | | CAC ÷ (ARPU × gross margin), в месяцах |

```

## 7. ПЛАТФОРМЫ ПРОДУКТА [Core]

> Современный продукт редко живёт на одной платформе. Этот раздел определяет, **где именно живёт продукт** и **как платформы дополняют друг друга**.

### 7.1 Чек-лист платформ

> **Обязательная база курса (отмечено ✅):** web + мобильное приложение. Остальное — опционально, поверх базы.

| Платформа | Статус | Роль | Когда нужна |
|-----------|--------|------|-------------|
| **Web** (десктоп + мобильный браузер) | ✅ обязательно | primary / companion / landing | основной кейс — сложные интерфейсы, SEO, share по ссылке |
| **Мобильное приложение** (iOS и/или Android) | ✅ обязательно | primary / companion | пуши, камера, в кармане, премиум-/массовая аудитория |
| Telegram-бот | ☐ опционально | primary / companion / acquisition | быстрые сценарии, нет онбординга, низкий CAC в СНГ |
| WhatsApp-бот | ☐ опционально | primary / companion | глобальный рынок, бизнес-кейсы |
| Chrome-расширение | ☐ опционально | primary / companion | контекстные действия в чужом UI (LinkedIn, Notion, Figma) |
| Figma plugin | ☐ опционально | primary / companion | для дизайн-инструментов |
| Desktop (Mac/Win) | ☐ опционально | primary / companion | работа с файлами, фоновые задачи, hotkeys |
| Voice (Alexa, голос) | ☐ опционально | companion | разговорные сценарии |
| Email-only | ☐ опционально | companion | дайджесты, нотификации |
| Другое: ___________ | ☐ опционально | | |

**Роли:**
- **Primary** — основное место создания ценности. Здесь происходит главный JTBD.
- **Companion** — поддерживает primary (нотификации, быстрый ввод, viewer).
- **Acquisition** — канал привлечения, пользователь сюда сначала попадает (например, бот для квиза → ведёт в web-приложение).

> **Решение по базе:** определи, какая из двух обязательных платформ (web или mobile) у тебя **primary** — то есть несёт основную ценность, а какая **companion** — дополняет. Это определит, куда идёт основной MVP-фокус.

### 7.2 Кросс-платформенный сценарий

> Опиши один путь пользователя, в котором задействованы 2+ платформы. Если все платформы изолированы — возможно, это не один продукт, а несколько.
