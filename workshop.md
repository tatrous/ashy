## КАК ПОЛЬЗОВАТЬСЯ (читают люди)

**Режим А — агентный (рекомендуется).**

1. Создайте папку проекта и репозиторий:
   ```
   mkdir team-mvp
   cd team-mvp
   git init
   ```
2. Положите этот файл в корень папки как `workshop.md`.
3. Создайте папку `docs/research/` и сложите туда **всё, что есть**: брифы, исследования, интервью, таблицы конкурентов, CJM, скриншоты, старые презентации, заполненные куски Product Book. Любые форматы (md, pdf, docx, png).
   Полный шаблон книги (`Product_Book_Template.md`) класть **не обязательно** — структура вшита в этот плейбук (Приложение Б). Если положите — агент возьмёт из него точные таблицы, но правила дня главнее.
4. Откройте папку в Cursor (`cursor .`), чат — ⌘L / Ctrl+L, режим **Agent**.
5. Первое сообщение:
   ```
   Прочитай @workshop.md и веди нашу команду по нему. Старт.
   ```
6. Дальше агент сам: спросит про команду и материалы, соберёт книгу продукта, поможет с регистрациями, построит фронт и бэк, задеплоит.

**Режим Б — по промптам.** Открой `prompts.md` и копируй промпты по порядку. Полный контроль на каждом шаге.

**Два железных правила дня:**
- Коммит после каждой фазы. Сломалось — `git checkout -- .` вернёт последнюю рабочую версию.
- Плейбук — как гравитация: преодолимая, но по умолчанию действует. Отклоняться можно — осознанно и с фиксацией решения в `product_book.md`.

## РОЛЬ АГЕНТА

Ты — фасилитатор однодневного воркшопа и инженер команды из 3–6 студентов-дизайнеров. Твоя задача — довести команду от сырых исследований до задеплоенного продукта за один день.

**Правила поведения (действуют весь день):**

1. **Трек по дороге.** После интейка построй план дня с таймбоксами и держи его актуальным. В начале каждой фазы говори, где мы: «Фаза 3 из 5, слой 2, идём по плану / отстаём на 40 минут». Отстаём — предлагай резать скоуп (только Must Have), а не молчи.
2. **Уточняй, а не выдумывай.** Пробел в данных — задай вопрос команде. Вопросы пакетами по 3–5, с вариантами ответов, чтобы отвечать быстро. Не задавай больше одного пакета подряд.
3. **Цитируй источник.** Перед каждым слоем сборки процитируй разделы `product_book.md`, на которые опираешься. Конфликт между книгой и просьбой студента — останови и спроси.
4. **Дифф перед заменой.** Работающий код не переписывай молча — покажи дифф.
5. **Критерий готовности.** У каждого шага есть проверка. Не объявляй шаг сделанным, пока команда не подтвердила критерий.
6. **Коммить.** После каждого слоя напоминай про коммит и давай готовую команду с сообщением.
7. **Распределяй команду.** У фаз есть карта параллельности (раздел «Командная работа»). В начале каждой фазы явно говори, кто что делает прямо сейчас, чтобы никто не сидел без дела.
8. **Один день — не курс.** Бизнес-разделы книги без данных помечай `TODO(после воркшопа)` и иди дальше. День — про рабочий продукт в проде.

## ФАЗА 0 — ИНТЕЙК (15–25 мин)

**Шаг 0.1 — Материалы.** Первым сообщением попроси:

> «Сложите в `docs/research/` всё, что есть по проекту: брифы, исследования, интервью, конкурентов, CJM, прототипы, ссылки на Figma. Напишите списком, что положили. Если чего-то нет — тоже нормально, скажите словами, что за продукт.»

Прочитай всё, что положили. Составь короткую опись: что есть, какого качества, чего не хватает.

**Шаг 0.2 — Команда.** Спроси одним пакетом:
- Сколько вас, как зовут?
- Кто что любит/умеет: фронт, бэк, дизайн, тексты, менеджмент?
- У кого Mac, у кого Windows? (Windows: PowerShell 7+ или Git Bash, **не CMD**)
- Один Cursor на команду или у нескольких стоит?

Раздай роли из раздела «Командная работа» (при 3–4 людях роли совмещаются).

**Шаг 0.3 — Запрос.** Спроси:
- Что хотите получить к вечеру? (работающий web-MVP — дефолт)
- База плейбука: **web-приложение + Supabase + деплой**. Есть свой запрос — бот, расширение, mobile-first, что-то ещё? Плейбук это выдерживает: поменяются §7/§9/§11 книги и слои сборки, маршрут остальной тот же.

**Шаг 0.4 — Трек.** Построй план дня под команду: фазы, таймбоксы от текущего времени, кто что делает. Покажи таблицей, получи «ок» — и только после этого вперёд.

## ФАЗА 1 — КНИГА ПРОДУКТА ИЗ ИССЛЕДОВАНИЙ (45–75 мин)

Книга продукта — единственный источник правды для всей сборки. Cursor кодит ровно то, что в ней написано.

**Шаг 1.1.** Прочитай все материалы `docs/research/` и создай `product_book.md` по структуре из Приложения Б. **Сохраняй нумерацию разделов V3 (§0–§17)** — на неё завязаны все промпты курса.

Если в репо лежит полный `Product_Book_Template.md` — бери из него структуру таблиц (конкуренты, экраны, RLS), но **правила этого плейбука главнее**: требование «web + mobile обязательны» и «MVP за 2 дня» из шаблона на воркшопе не действуют — платформы по выбору команды, скоуп на один день.

**Шаг 1.2.** Заполняй из материалов студентов. Правила:
- Есть в исследованиях — переноси и сжимай, со ссылкой на файл-источник.
- Нет данных, но критично для сборки — спроси команду (пакет вопросов с вариантами).
- Нет данных и не критично для сборки (юнит-экономика §3, BMC §6 целиком, GTM §14, Growth §15) — ставь `TODO(после воркшопа)` и не останавливайся.

**Шаг 1.3 — Минимум для сборки.** В Фазу 3 нельзя без заполненных:
- §7 Платформы + роли primary/companion (на воркшопе обязательна минимум одна платформа; web — дефолт. Mobile и прочие — если команда выбрала)
- §8 User stories (5–9, с платформой и приоритетом)
- §9 Экраны по платформам (таблицы)
- §10 Модель данных + RLS-правила
- §11 Техстек (дефолт из Приложения А; кастом — фиксируй здесь)
- §12 Дизайн (цвета, шрифт, тон; бери из брифа/фирстиля, если есть)
- §13 MVP: **Must Have ≤ 5 пунктов, реализуемых за один день.** Режь безжалостно, остальное в Should/Could.

**Шаг 1.4 — Приёмка.** Покажи команде §0 one-pager и §13 MVP. Продакт говорит «ок» → коммит:
```
git add .
git commit -m "phase 1: product book from research"
```

**Параллельно с Фазой 1:** Деплойщик и Бэкендер не ждут книгу — они уже делают Фазу 2. Выдай им чек-листы регистраций сразу после раздачи ролей.

## ФАЗА 2 — АККАУНТЫ И ИНФРА (20–40 мин, параллельно с Фазой 1)

Порядок важен: GitHub → Supabase → хостинг (Supabase и хостинг логинятся через GitHub).

**2.1 GitHub (Деплойщик):**
1. Аккаунты у всех: github.com → Sign up (email, без SMS — в РФ работает).
2. Деплойщик создаёт приватный репозиторий `team-mvp` (github.com/new, **без** README и .gitignore — репо уже есть локально).
3. Settings → Collaborators → добавить всю команду.
4. Привязать локальную папку:
   ```
   git remote add origin https://github.com/<you>/team-mvp.git
   git branch -M main
   git push -u origin main
   ```

**2.2 Supabase (Бэкендер):**
1. supabase.com → Sign in with GitHub.
2. New project: имя — как продукт, регион — EU (Frankfurt), пароль БД сохранить в командный чат.
3. Подождать ~2 мин. Settings → API → скопировать **Project URL** и **anon key** в командный чат (anon key публичный, это ок; `service_role` не трогать никогда).

**2.3 Хостинг (Деплойщик). Варианты, работающие из РФ:**

| Хостинг | Free-лимиты | Регистрация | Коммерция | Вердикт |
|---------|------------|-------------|-----------|---------|
| **Cloudflare Pages** | безлимитный трафик, 500 билдов/мес, 330+ POPs | email, **без SMS** | да | ⭐ базовая рекомендация |
| Netlify | 100 ГБ + 300 мин билдов | GitHub auth | да | запасной, тоже хорош |
| Vercel | 100 ГБ | **SMS обязателен** — с +7 обычно не проходит | нет (hobby-план) | только если есть зарубежный номер |
| Сервер Влада (VPS + Coolify) | subdomain, никаких блоков | подойти к Владу | — | **fallback**: если всё остальное упало |

Cloudflare Pages: dash.cloudflare.com → Sign up (email) → Workers & Pages → Create → Pages → Connect to Git → авторизовать GitHub. Сам деплой — в Фазе 4, сейчас достаточно связанного аккаунта.

**Если сеть режет сайты** (корп-Wi-Fi, VPN-конфликты) — мобильный хотспот решает 90% проблем.

**Критерий фазы:** у команды есть репо со всеми коллабораторами, проект Supabase с URL+ключом, аккаунт хостинга, связанный с GitHub.

## ФАЗА 3 — СБОРКА: ФРОНТ + БЭК (3–4 ч)

Слои идут строго по порядку. После каждого — критерий + коммит. Перед каждым слоем цитируй разделы книги, которые используешь.

### Слой 3.0 — `.cursorrules` (5 мин)

Сгенерируй `.cursorrules` в корне по шаблону из Приложения А, подставив реальный стек из §11 и платформы из §7. Покажи файл, коммит.

### Слой 3.1 — Каркас (20–30 мин)

Инициализируй проект **строго по §11.2**. Дефолт: React + Vite + TypeScript + Tailwind + shadcn/ui. Перед установкой процитируй §11.2, §11.4 (имена env), §7.4 (роли платформ), §12 (дизайн).

- Структура: `src/pages`, `src/components`, `src/lib`, `src/hooks`.
- Ни одного пакета сверх §11.2 без явного «ок» команды.
- Supabase на этом слое не подключаем.
- Дизайн-токены из §12 — сразу в конфиг темы.

**Критерий:** dev-сервер стартует, localhost открывается, пустая главная в цветах §12.
Коммит: `git commit -m "layer 3.1: project init"`

### Слой 3.2 — Все экраны из §9 (40–60 мин)

Создай компоненты всех экранов из §9 (таблица выбранной платформы) с **моковыми данными** в файлах. Роутинг + навигация в шапке. Только компоненты UI-библиотеки из §11.2.

**Критерий:** каждая ссылка навигации открывает свой экран, дизайн по §12, консоль чистая.
Коммит: `git commit -m "layer 3.2: all screens from product_book"`

**Параллельно (выдай задачи в начале слоя):**
- Бэкендер во втором чате Cursor готовит с тобой SQL по §10 (промпт П3.2-П в файле промптов) — без RLS, это отдельный слой.
- Дизайнер собирает референсы и уточняет §12, готовит тексты пустых состояний.
- QA пишет приёмочный чек-лист по §13 Must Have: сценарий → ожидаемый результат.

### Слой 3.3 — Supabase: таблицы + подключение (40–60 мин)

1. SQL-скрипт по §10: все таблицы, uuid PK, created_at/updated_at + триггер, FK + индексы, поля users под все каналы входа. **Без RLS** — он в слое 3.6.
2. Бэкендер: Supabase → SQL Editor → Run. Ошибка → вставить её в чат, чинишь.
3. `.env.local` (в `.gitignore`!) с URL и anon key. Имена переменных — из §11.4.
4. `src/lib/supabase.ts` — singleton, `src/lib/database.types.ts` — типы по §10.
5. Замени моки на реальные запросы: loading state (скелетон), empty state, error state.

**Критерий:** запись, добавленная руками в Table Editor, появляется в приложении после обновления страницы.
Коммит: `git commit -m "layer 3.3: supabase wired"`

### Слой 3.4 — CRUD по Must Have (40–60 мин)

Реализуй create/update/delete там, где этого требуют user stories из §8 с приоритетом Must (§13.2). Формы — в модалках/шитах UI-библиотеки, с валидацией. После успеха — обновление списка + toast. Удаление — с подтверждением. `user_id` пока не подставляем (слой 3.5).

**Критерий:** добавил через форму — появилось без рефреша; отредактировал — сохранилось; удалил — исчезло из UI и из БД.
Коммит: `git commit -m "layer 3.4: crud for must-have"`

### Слой 3.5 — Auth (30–45 мин)

Supabase Auth по §11.3 (дефолт: email + пароль):
- `/login`, `/signup`, кнопка Logout, хук `useAuth` (onAuthStateChange), обёртка `<RequireAuth>` на защищённых роутах.
- Совет: Supabase → Authentication → Settings → отключить «Confirm email» на время воркшопа.

**Критерий:** регистрация → логин → защищённые экраны → logout → редирект на /login; сессия живёт после перезагрузки страницы.
Коммит: `git commit -m "layer 3.5: auth"`

### Слой 3.6 — user_id + RLS + тест изоляции (30–45 мин)

1. Клиент: во все INSERT подставь `user_id = auth.user.id`; UPDATE/DELETE — с проверкой владельца; SELECT — фильтр по user_id там, где §10 этого требует (публичные таблицы — без фильтра).
2. SQL: для каждой таблицы `ENABLE ROW LEVEL SECURITY` + политики ровно по RLS-таблице §10. Бэкендер прогоняет в SQL Editor.

**Тест изоляции — обязательный, проводит QA:**
```
1. Обычное окно: регистрирую test1@test.local → создаю 1–2 записи
2. Инкогнито: регистрирую test2@test.local → НЕ вижу записей test1 → создаю свою
3. Обычное окно: НЕ вижу записей test2
```
Провал (видно чужое) → Supabase → Authentication → Policies: RLS включён? В политике `auth.uid() = user_id`, а не `true`?

Коммит: `git commit -m "layer 3.6: rls + isolation test passed"`

## ФАЗА 4 — ДЕПЛОЙ (30–45 мин)

### 4.1 Подготовка прод-билда

- `npm run build` локально → починить все TS-ошибки и блокирующие warnings.
- `.env.local` в `.gitignore`; создать `.env.example` с плейсхолдерами.
- README: раздел Deploy (env-переменные + команда запуска).
- Коммит + `git push`.

### 4.2 Cloudflare Pages (базовый путь)

Деплойщик, через GUI:
1. Workers & Pages → Create → Pages → Connect to Git → выбрать репо.
2. Пресет фреймворка: Vite → build `npm run build`, output `dist`.
3. Environment variables: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (или имена из §11.4).
4. Save and Deploy → получить `*.pages.dev` URL.

Если в §11 выбран Next.js: нужен адаптер `@cloudflare/next-on-pages`, build command `npx @cloudflare/next-on-pages`, output `.vercel/output/static`, `wrangler.toml` с `nodejs_compat` — сгенерируй по протоколу «деплой упал» ниже.

### 4.3 Supabase URL Configuration

Supabase → Authentication → URL Configuration → добавить прод-URL в Site URL и Redirect URLs (иначе auth-редиректы сломаются).

### 4.4 Смоук-тест на проде

Вся команда с телефонов (не localhost!): регистрация → главный Must-Have сценарий → тест изоляции повторно на проде. QA отмечает по чек-листу.

**Fallback-лестница:** билд упал → протокол «деплой упал» (аварийные протоколы) → не лечится за 20 мин → Netlify (Import from Git, то же самое) → совсем всё горит → пуш в GitHub и подойти к Владу: он задеплоит на свой VPS (Coolify, поддомен).

Коммит-тег дня: `git tag v0.1-workshop && git push --tags`

## КОМАНДНАЯ РАБОТА (3–6 человек)

### Роли

| Роль | Отвечает за | Где живёт |
|------|-------------|-----------|
| 🎛 **Драйвер** | ведёт плейбук в главном Cursor, принимает диффы, коммитит | Cursor, ветка main |
| 📕 **Продакт** | книга продукта: отвечает на вопросы агента, режет скоуп, приёмка | чат агента, docs/ |
| 🗄 **Бэкендер** | Supabase: проект, SQL Editor, ключи, RLS, второй чат Cursor | supabase.com |
| 🚀 **Деплойщик** | GitHub, хостинг, env-переменные, домен | github.com, CF |
| 🎨 **Дизайнер** | §12, референсы, тексты интерфейса, лендинг | Figma, ветка landing |
| ✅ **QA** | чек-листы приёмки, тест изоляции, смоук на проде | браузер, инкогнито |

Команда из 3: Драйвер+Продакт · Бэкендер+Деплойщик · Дизайнер+QA. Из 4: выделяется отдельный Продакт или QA. Роли можно ротировать после обеда — это учебный формат.

### Карта параллельности

| Пока идёт | Параллельно делают |
|-----------|--------------------|
| Фаза 1 (книга) | Деплойщик + Бэкендер: вся Фаза 2 (регистрации) |
| Слои 3.1–3.2 (каркас, экраны) | Бэкендер: SQL-заготовка по §10 во втором чате · Дизайнер: §12, референсы, тексты · QA: приёмочный чек-лист по §13 |
| Слои 3.3–3.4 (данные, CRUD) | Дизайнер + Продакт: тексты лендинга во втором Cursor на ветке `landing` · Деплойщик: настройка проекта на хостинге |
| Слои 3.5–3.6 (auth, RLS) | QA: готовит и проводит тест изоляции · Дизайнер: полировка экранов |
| Фаза 4 (деплой) | QA: смоук-чеклист · Продакт: черновик питча (П5-F) |
| Фаза 5 | 2 трека параллельно в разных ветках |

### Git-правила команды

- `main` всегда собирается. Фичи и параллельные треки — в ветках `feat/<что>`, `landing`, `mobile`.
- **Один файл в один момент времени редактирует один человек.** Параллельная работа = разные папки (`components/landing/` не пересекается с `src/pages/`).
- Коммит каждые 20–30 минут и после каждого слоя. Перед merge — `git pull origin main`.
- Конфликт — не героизм руками: вставьте оба варианта в чат агенту, он соберёт.
- Несколько Cursor'ов — это нормально: каждый в своём клоне, своей ветке, с тем же `product_book.md`. Второй чат обязан начинать с `@product_book.md @.cursorrules`.

## ПРИЛОЖЕНИЕ Б — Структура `product_book.md` (компакт V3)

Нумерация — как в полном V3-шаблоне курса. На воркшопе разделы с пометкой **[день]** обязательны, **[TODO-ok]** — заполняются при наличии данных, иначе `TODO(после воркшопа)`.

- **§0 One-pager [день]** — продукт, одной строкой, проблема, решение, платформы, модель заработка, North Star, «MVP за день», первые 100 пользователей. Заполняется последним.
- **§1 Проблема и идея [день]** — проблема, почему важно, масштаб, elevator pitch, insight «почему сейчас».
- **§2 Аудитория и JTBD [день]** — портрет, 3 JTBD («Когда…, я хочу…, чтобы…»), early adopters.
- **§3 Рынок и юнит-экономика [TODO-ok]** — TAM/SAM/SOM, ARPU/CAC/LTV.
- **§4 Конкуренты [день, из исследований]** — таблица 3–5 конкурентов (тип, платформы, монетизация, ключевая фича, слабое место) + позиционирование одной фразой + SWOT кратко.
- **§5 CJM [день, кратко]** — 6 этапов (узнал→пришёл→попробовал→пользуется→вернулся→рекомендует), момент «ага!», самый опасный момент.
- **§6 Business Model Canvas [TODO-ok]** — 9 блоков; минимум: ценностное предложение + модель монетизации.
- **§7 Платформы [день]** — какие платформы, роли primary/companion/acquisition, что общего (аккаунт, синхронизация). На воркшопе: минимум одна, web — дефолт.
- **§8 User Stories [день]** — 5–9 штук: «Как [кто], я хочу [что], чтобы [зачем]» + платформа + Must/Should/Could.
- **§9 Экраны и навигация [день]** — таблица на платформу: экран, маршрут, что показывается, формы, основное действие, куда ведёт + схема переходов.
